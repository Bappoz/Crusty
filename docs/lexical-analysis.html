Análise Léxica

Transformando texto bruto em uma sequência de tokens significativos — a primeira fase do pipeline de compilação do Rippler.

Implementada e documentada

O que é Análise Léxica? #

A análise léxica (ou scanning) é a primeira fase de um compilador. Seu trabalho é ler o código-fonte caractere por caractere e agrupar sequências significativas em unidades chamadas tokens.

Imagine que o compilador está lendo o trecho de código abaixo: 

int x = 42 + y;

O analisador léxico enxerga isso como uma sequência de seis tokens:

LexemaTipo de TokenCategoria
intIntPalavra-chave
xIdentifier("x")Identificador
=EqualOperador
42IntLiteral(42)Literal inteiro
+PlusOperador
yIdentifier("y")Identificador
;SemicolonPontuação

Espaços em branco e comentários são descartados durante esse processo. Os tokens resultantes são passados para o parser (Capítulo 02).

Arquitetura do Scanner #

O scanner do Rippler é implementado na struct Scanner em src/lexer/scanner.rs. Ela encapsula o estado completo da varredura:

src/lexer/scanner.rs 
pub struct Scanner {
    pub src: SourceFile,              // fonte sendo lida
    pub tokens: Vec<Token>,           // tokens produzidos
    pub diagnostics: Vec<CompilerError>, // erros coletados
    delimiter_stack: Vec<(char, usize, usize)>, // pilha de delimitadores abertos
}

SourceFile e Memory Mapping

A leitura do código-fonte é feita pelo SourceFile (src/common/input/source.rs), que utiliza memory mapping (via crate memmap2) para carregar o arquivo na memória de forma eficiente sem copiar bytes desnecessariamente.

src/common/input/source.rs — criação do SourceFile 
// A partir de arquivo em disco (usa mmap)
let src = SourceFile::from_path(PathBuf::from("main.c"))?;

// A partir de string (usado nos testes)
let src = SourceFile::from_string("int x = 42;");

O SourceFile rastreia três informações de posição durante a varredura:

  • pos — offset de bytes no buffer UTF-8
  • line — número da linha atual (começa em 1)
  • col — número da coluna atual (começa em 1)

ByteSpan

Cada token carrega um ByteSpan com os offsets de início e fim no buffer original, além de line e col. Isso permite que as mensagens de erro apontem com precisão para o local do problema no código-fonte.

src/common/input/span.rs 
pub struct ByteSpan {
    pub start: usize,
    pub end: usize,
}

pub struct Token {
    pub kind: TokenKind,
    pub span: ByteSpan,
    pub line: usize,
    pub col: usize,
}

Processo de Varredura #

O método scan() é o ponto de entrada da análise léxica. Ele executa um loop que alterna entre pular espaços/comentários e reconhecer o próximo token, até consumir todo o arquivo.

src/lexer/scanner.rs — método scan() 
pub fn scan(&mut self) -> &[Token] {
    while !self.src.is_at_end() {
        self.skip_whitespaces_and_comments();
        if self.src.is_at_end() { break; }
        self.next_token();
    }

    // Delimitadores ainda na pilha = nunca foram fechados
    let unclosed: Vec<_> = self.delimiter_stack.drain(..).collect();
    for (c, line, col) in unclosed {
        self.diagnostics.push(/* UnclosedDelimiter(c) */);
    }

    // Sempre termina com EOF para o parser
    self.emit_at(TokenKind::Eof, "", self.src.line(), self.src.col());
    &self.tokens
}

O método next_token() lê um caractere com src.advance() e usa um match para despachar para o handler correto:

src/lexer/scanner.rs — next_token() (simplificado) 
fn next_token(&mut self) {
    let (line, col) = (self.src.line(), self.src.col());
    let c = self.src.advance().unwrap();

    match c {
        '0'..='9'           => self.lex_number(c, line, col),
        '"'                  => self.lex_string(line, col),
        '\''                 => self.lex_char(line, col),
        c if is_ident_start(c) => self.lex_identifier(c, line, col),
        '(' | '[' | '{'      => { /* empilha + emite */ }
        ')' | ']' | '}'      => { /* desempilha + emite */ }
        '+' | '-' | '*' | '/' | '=' | '!' | '<' | '>' | '&' | '|'
                             => self.lex_operator(c, line, col),
        ';' | ',' | ':' | '.' | '%' | '^' | '~'
                             => self.emit_at(/* token simples */),
        c                    => self.emit_unknown(c, line, col),
    }
}

Espaços em Branco e Comentários #

Antes de cada token, skip_whitespaces_and_comments() consome silenciosamente três categorias de conteúdo irrelevante para a semântica:

Espaços em branco

Caracteres ' ', '\t', '\r' e '\n' são descartados em loop. Quebras de linha incrementam automaticamente o contador de linha no SourceFile.

Comentários de linha (//)

Ao detectar // (dois / consecutivos via peek() + peek_ahead()), o scanner avança até encontrar '\n' ou EOF.

Comentários de bloco (/* ... */)

Ao detectar /*, o scanner consome caracteres até encontrar */. Se o EOF for atingido antes do fechamento, é emitido um diagnóstico UnclosedBlockComment e a varredura continua.

Diretivas de pré-processador (#)

Linhas que começam com # (como #include e #define) são ignoradas integralmente. O pré-processador não é implementado no Rippler.

src/lexer/scanner.rs — skip_whitespaces_and_comments() 
fn skip_whitespaces_and_comments(&mut self) {
    loop {
        // Pula espaços
        while matches!(self.src.peek(), Some(' '|'\t'|'\r'|'\n')) {
            self.src.advance();
        }

        // Comentário de linha: //
        if self.src.peek() == Some('/') && self.src.peek_ahead() == Some('/') {
            while !matches!(self.src.peek(), Some('\n') | None) {
                self.src.advance();
            }
            continue;
        }

        // Comentário de bloco: /* ... */
        if self.src.peek() == Some('/') && self.src.peek_ahead() == Some('*') {
            // ... consome até '*/' ou EOF com diagnóstico
            continue; // se fechado; break se EOF
        }

        // Diretivas de pré-processador: #...
        if self.src.peek() == Some('#') {
            while !matches!(self.src.peek(), Some('\n') | None) {
                self.src.advance();
            }
            continue;
        }

        break;
    }
}

Identificadores e Palavras-chave #

Um identificador começa com uma letra ou _ (verificado por is_ident_start(c)), seguido por zero ou mais letras, dígitos ou _ (is_ident_continue(c)).

src/lexer/rules/identifiers.rs 
fn lex_identifier(&mut self, first: char, line: usize, col: usize) {
    let mut buf = String::from(first);

    while let Some(c) = self.src.peek() {
        if is_ident_continue(c) {
            buf.push(c);
            self.src.advance();
        } else {
            break;
        }
    }

    // Verifica se é uma keyword ou identificador de usuário
    let kind = lookup_keyword(&buf).unwrap_or(TokenKind::Identifier(buf.clone()));
    self.emit_at(kind, &buf, line, col);
}

A função lookup_keyword() faz o match do lexema contra a tabela completa de palavras reservadas do C. Se não encontrar correspondência, o token é Identifier(buf).

Design: keywords como casos do TokenKind

No Rippler, cada keyword tem sua própria variante no enum TokenKind (ex: TokenKind::If, TokenKind::While). Isso elimina a necessidade de string comparisons no parser — o match arm já é o tipo exato.

Literais #

As regras de literais estão em src/lexer/rules/literals.rs e cobrem inteiros (decimal, octal, hexadecimal), floats, strings e chars.

Literais Inteiras

O scanner detecta automaticamente a base numérica pelo prefixo:

0x / 0XHexadecimal (16)0xFF255
0 + dígito octalOctal (8)0755493

src/lexer/rules/literals.rs — lex_number() 
fn lex_number(&mut self, first: char, line: usize, col: usize) {
    let mut buf = String::from(first);

    // Hexadecimal: 0x...
    if first == '0' && matches!(self.src.peek(), Some('x') | Some('X')) {
        buf.push(self.src.advance().unwrap()); // 'x'
        while let Some(c) = self.src.peek() {
            if is_hex_digit(c) { buf.push(c); self.src.advance(); } else { break; }
        }
        let value = i64::from_str_radix(&buf[2..], 16).unwrap_or(0);
        return self.emit_at(TokenKind::IntLiteral(value), &buf, line, col);
    }

    // Octal: 0755
    if first == '0' && matches!(self.src.peek(), Some('0'..='7')) {
        while let Some(c) = self.src.peek() {
            if is_octal_digit(c) { buf.push(c); self.src.advance(); } else { break; }
        }
        let value = i64::from_str_radix(&buf[1..], 8).unwrap_or(0);
        return self.emit_at(TokenKind::IntLiteral(value), &buf, line, col);
    }

    // Decimal ou Float
    // ... consome dígitos, parte fracionária (.xxx) e expoente (eXXX)
}

Literais Float

Se após os dígitos decimais o scanner encontrar ., e ou E, a literal é tratada como float. A parte fracionária e o expoente (com sinal opcional +/-) são consumidos e o resultado é parseado como f64.

FormaExemploToken gerado
Inteiro decimal100IntLiteral(100)
Com ponto3.14FloatLiteral(3.14)
Com expoente1e10FloatLiteral(1e10)
Com sinal no exp.1.5e-3FloatLiteral(0.0015)

Literais de String

Strings são delimitadas por "...". O scanner processa sequências de escape (\n, \t, \\, etc.) via resolve_escape() e armazena o conteúdo processado no token. Se o arquivo terminar antes do " de fechamento, é emitido um diagnóstico UnterminatedLiteral.

Literais de Char

Chars são delimitados por '...' e suportam os mesmos escapes das strings. Apenas um caractere (ou um escape) é esperado entre as aspas simples.

Limitação atual

A versão atual não valida que a literal de char contém exatamente um caractere — chars multi-char como 'ab' podem produzir comportamento inesperado. Isso será tratado na fase de análise semântica.

Operadores #

O scanner usa lookahead de um caractere para distinguir operadores simples de compostos. Ao encontrar um dos chars de operador (+ - * / = ! < > & |), chama lex_operator() em src/lexer/rules/operators.rs.

src/lexer/rules/operators.rs — exemplo para '+' e '<' 
'+' => match self.src.peek() {
    Some('+') => { self.src.advance(); self.emit_at(TokenKind::PlusPlus,  "++", line, col); }
    Some('=') => { self.src.advance(); self.emit_at(TokenKind::PlusEqual, "+=", line, col); }
    _         =>                       self.emit_at(TokenKind::Plus,       "+",  line, col),
},

'<' => match self.src.peek() {
    Some('=') => { self.src.advance(); self.emit_at(TokenKind::LessEqual, "<=", line, col); }
    Some('<') => {
        self.src.advance();
        // Lookahead extra para <<=
        if self.src.peek() == Some('=') {
            self.src.advance();
            self.emit_at(TokenKind::LessLessEqual, "<<=", line, col);
        } else {
            self.emit_at(TokenKind::LessLess, "<<", line, col);
        }
    }
    _ => self.emit_at(TokenKind::Less, "<", line, col),
},
Nota: // e /*

Os tokens / e /* são tratados especialmente: comentários são consumidos em skip_whitespaces_and_comments() antes de next_token(). Por isso, quando lex_operator('/') é chamado, só chegam / isolado ou /=.

Delimitadores e Pareamento #

O Rippler rastreia o pareamento de delimitadores ( ) [ ] { } durante a própria varredura léxica, usando uma pilha interna (delimiter_stack).

Ao encontrar um delimitador de abertura, o char e sua posição são empilhados. Ao encontrar um de fechamento, o topo da pilha é verificado:

  • Se o par bate, desempilha normalmente.
  • Se não bate (ou a pilha está vazia), emite UnexpectedClosingDelimiter.
  • Ao final do arquivo, quaisquer entradas restantes na pilha viram UnclosedDelimiter.
src/lexer/scanner.rs — tratamento de delimitadores 
// Abertura: empilha e emite
'(' => {
    self.delimiter_stack.push(('(', line, col));
    self.emit_at(TokenKind::LeftParen, "(", line, col);
}

// Fechamento: valida o par
')' => {
    if matches!(self.delimiter_stack.last(), Some(('(', _, _))) {
        self.delimiter_stack.pop();
    } else {
        self.emit_unexpected_delimiter(')', line, col);
    }
    self.emit_at(TokenKind::RightParen, ")", line, col);
}

// Ao final do scan():
let unclosed = self.delimiter_stack.drain(..).collect::>();
for (c, line, col) in unclosed {
    // UnclosedDelimiter(c) para cada um
}
Por que detectar isso no léxico?

Embora o pareamento de delimitadores seja tecnicamente responsabilidade do parser, detectá-lo no léxico permite emitir mensagens de erro muito mais precisas: sabemos exatamente em qual linha e coluna o delimitador foi aberto e onde o problema foi detectado.

Tratamento de Erros #

Erros léxicos são representados pelo enum LexicalErrorKind e coletados em scanner.diagnostics. O scanner nunca para ao encontrar um erro — ele registra e continua, permitindo que vários erros sejam reportados em uma única execução.

Tipos de erro léxico

VarianteSituaçãoExemplo
InvalidChar(char) Caractere que o scanner não reconhece @, $, `
UnclosedBlockComment /* aberto sem */ antes do EOF /* comentário...
UnclosedDelimiter(char) Abertura sem fechamento até o final do arquivo (foo + bar
UnexpectedClosingDelimiter(char) Fechamento sem abertura correspondente x + y)
UnterminatedLiteral(String) String ou char sem fechamento antes do EOF "hello, 'a

Sistema de relatório de erros

Cada CompilerError pode ser convertido em um Report estruturado via o trait ToReport. Um Report carrega: mensagem principal, span do erro, label inline e sugestão de correção (help).

Exemplo de diagnóstico gerado 
error: unclosed delimiter
  --> main.c:3:5
   |
 3 |     (x + y
   |     ^ '(' nao foi fechado
   |
   = help: Adicione o delimitador de fechamento correspondente.
